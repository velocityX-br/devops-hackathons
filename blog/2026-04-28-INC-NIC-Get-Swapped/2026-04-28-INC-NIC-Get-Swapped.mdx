## Introduction

This situation perfectly illustrates the risks that **udev** rules and inappropriate **Netowrk Interface** operation (From CloudProvider/Openstack) pose to VM stability. It all began with a customer report stating their VM failed to boot after a routine flavor upgrade. 

{/* truncate */}

## Symptoms and investigation

Dmesg shows the NIC name has been changed, also eth0 can't be found by wicked journaling. Link is down.
![Dmesg vmxnet changed NIC name](./vmxnet_renamed_eth0.jpeg)

```
[    5.609082] [    T977] vmxnet3 0000:0b:00.0 eth0: NIC Link is Up 10000 Mbps
[    5.616024] [    T977] vmxnet3 0000:13:00.0 eth1: NIC Link is Up 10000 Mbps
[    5.619816] [    T977] vmxnet3 0000:1b:00.0 eth2: NIC Link is Up 10000 Mbps
[    5.633900] [    T967] vmxnet3 0000:0b:00.0 rename2: renamed from eth0

[    5.749051] [    T967] vmxnet3 0000:0b:00.0 eth1: renamed from rename2
```

![GuestOS Wicked Journaling](./wicked_state.png)


At the very beginging, I thought it is a driver issue therefore engaged the hypervisor team to check the driver, including EXSi version versus network driver (Mellanox, VMware) or firmware version coz there had such incidents before. 

![NIC Link Down](./NIC_Link_Down.png)
As seens from the `ip link`, obviously the eth0 is screwing up which have been renamed into rename2. Then I tried to fix the NIC name `ip link set rename2 name eth0`.
Then followed by a `wicked` restart. It is started ! But still no SSH/ICMP ping which indicates the NIC is still not working.
  - Ping gateway is not working. Gateway isn't screwing up. E.g `ip route list` or `ip route get <IP>`
  - `Arp -a` returns single uncompleted entry, instead of partially broken. E.g Successfully learn MAC address other other LAN E.g StorageLAN/BackupLAN. I tried to cross `tcpdump` check with `tcpdump -i eth0 -nn arp`
  - Honestly, I didn't check udev rules carefully at the beginning. `/etc/udev/rules.d/*/`  Note: higher number takes higher priority. (Note: this is common in Linux, similar to `/etc/sysctl.d/*/` files)

![TcpdumpArp_UdevRules](./TcpdumpArp_UdevRule.png) 

Evidently, layer2 is broken. We must fix layer2 first before layer3.
Meanwhile, my colleagues in Europe gave me a clue that MAC address could be changed in CloudProvider/Openstack. 
Then I tried to compare the Mac address in CloudProvider/Openstack and VM.

### Root cause

__Core Problem Observed!!__
It turned out to be the VM's NIC Mac address is loaded incorrectly. We must check udev rules carefully again then.

## Resolution
The final resolution is to correct 85-cloud-init rules to match the new MAC address in CloudProvider/Openstack, followed by a VM restart.
Note: Please also double check PCI slot number's MAC address via `lscpi -vvv` to gurantee the consistency.

## FAQ
Q:  While changing the NIC from Sap Cloud Infrastruture/OpenStack, what will happen to the PCI devices of the VM?

A:

```
In ESXi, each VMware vmxnet driver will hang with its fixed PCI SlotPCI Slot（虚拟硬件拓扑）. 
PCI Slot 

It is like the first, second PCIe slot on the motherboard.
0000:0b:00.0
0000:13:00.0
0000:1b:00.0
```

Q: How Linux VM detect vNIC change ？

A: Linux kernel detect in order: PCI address -> Load driver -> Create/Register Net Device
```
VMware/OpenStack
        ↓
vNIC（MAC）
        ↓
VMXNET3 PCI Device（0b:00.0）
        ↓
Linux driver
        ↓
eth0/enp0s3
```

Q: 在openstack 中修改网卡的mac地址后。lspci 仍然显示老的网卡信息, 如何解决？
```
0b:00.0 Ethernet controller: VMware VMXNET3 Ethernet Controller (rev 01) 
DeviceName: Ethernet0 
Subsystem: VMware VMXNET3 Ethernet Controller 
Physical Slot: 192 
...
Capabilities: [100 v1] Device Serial Number fa-16-3e-ff-ff-42-07-6d
```

A: 修改 OpenStack 里的 MAC 地址，不会改变 PCI 设备本身。 热插拔基本不可行。 请详细查看 `readlink /sys/class/net/eth0/device`
```
(vadbvaz) cc02v014382:/etc/udev/rules.d #
# readlink /sys/class/net/eth0/device
../../../0000:13:00.0
```
