## 引言

本案例很好地说明了 **udev** 规则以及云厂商/OpenStack 侧不当的 **网卡（Network Interface）** 操作对虚拟机稳定性的风险。一切始于客户报告：在一次常规的 flavor 升级后，虚拟机无法启动。

## 现象与排查

`dmesg` 显示网卡名称已被更改，且 wicked 日志中找不到 `eth0`，链路处于 down 状态。

![Dmesg：vmxnet 网卡名称被更改](./../../../../blog/2026-04-28-INC-NIC-Get-Swapped/vmxnet_renamed_eth0.jpeg)

```
[    5.609082] [    T977] vmxnet3 0000:0b:00.0 eth0: NIC Link is Up 10000 Mbps
[    5.616024] [    T977] vmxnet3 0000:13:00.0 eth1: NIC Link is Up 10000 Mbps
[    5.619816] [    T977] vmxnet3 0000:1b:00.0 eth2: NIC Link is Up 10000 Mbps
[    5.633900] [    T967] vmxnet3 0000:0b:00.0 rename2: renamed from eth0
[    5.699184] [    T977] vmxnet3 0000:13:00.0 eth0: renamed from eth1
[    5.749051] [    T967] vmxnet3 0000:0b:00.0 eth1: renamed from rename2
```

![GuestOS：Wicked 日志状态](./../../../../blog/2026-04-28-INC-NIC-Get-Swapped/wicked_state.png)

起初我怀疑是驱动问题，于是请虚拟化团队检查驱动，包括 ESXi 版本与网卡驱动（Mellanox、VMware）或固件版本的匹配——此前曾有过类似事件。

![网卡链路 Down](./../../../../blog/2026-04-28-INC-NIC-Get-Swapped/NIC_Link_Down.png)

从 `ip link` 可以明显看出 `eth0` 已被重命名为 `rename2`。我尝试修复网卡名称：`ip link set rename2 name eth0`，然后重启 `wicked`。服务起来了，但仍无法 SSH/ICMP ping，说明网卡仍未正常工作。

- 无法 ping 通网关。网关本身没问题，可用 `ip route list` 或 `ip route get <IP>` 确认。
- `arp -a` 只返回一条未完成的条目，而不是部分损坏。例如应能学到 StorageLAN/BackupLAN 上其他主机的 MAC。我用 `tcpdump -i eth0 -nn arp` 做了交叉验证。
- 老实说，一开始没有仔细核对 udev 规则：`/etc/udev/rules.d/*/`（注意：数字越大优先级越高，与 `/etc/sysctl.d/*/` 类似。）

![Tcpdump ARP 与 Udev 规则](./../../../../blog/2026-04-28-INC-NIC-Get-Swapped/TcpdumpArp_UdevRule.png)

显然二层已损坏，必须先修复二层再谈三层。

与此同时，欧洲的同事提示：云厂商/OpenStack 中的 MAC 地址可能已被修改。于是我对比了 OpenStack 与虚拟机内的 MAC 地址。

### 根因

**核心问题确认：**

虚拟机的网卡 MAC 地址被错误加载。必须再次仔细核对 udev 规则。

## 解决方案

最终方案是：将 `85-cloud-init` 规则中的 MAC 地址修正为与云厂商/OpenStack 中一致的新 MAC，然后重启虚拟机。

注意：请用 `lspci -vvv` 再次核对 PCI 槽位与 MAC 的对应关系，确保一致。

## 常见问题

**问：** 在 SAP Cloud Infrastructure/OpenStack 中更换网卡时，虚拟机的 PCI 设备会怎样？

**答：**

```
在 ESXi 中，每个 VMware vmxnet 驱动会绑定固定的 PCI Slot（虚拟硬件拓扑）。

PCI Slot 类似主板上的第一、第二块 PCIe 插槽：
0000:0b:00.0
0000:13:00.0
0000:1b:00.0
```

**问：** Linux 虚拟机如何感知 vNIC 变更？

**答：** Linux 内核按以下顺序检测：PCI 地址 → 加载驱动 → 创建/注册网络设备

```
VMware/OpenStack
        ↓
vNIC（MAC）
        ↓
VMXNET3 PCI Device（0b:00.0）
        ↓
Linux driver
        ↓
eth0/enp0s3
```

**问：** 在 OpenStack 中修改网卡的 MAC 地址后，`lspci` 仍显示旧的网卡信息，如何解决？

```
0b:00.0 Ethernet controller: VMware VMXNET3 Ethernet Controller (rev 01)
DeviceName: Ethernet0
Subsystem: VMware VMXNET3 Ethernet Controller
Physical Slot: 192
...
Capabilities: [100 v1] Device Serial Number fa-16-3e-ff-ff-42-07-6d
```

**答：** 修改 OpenStack 里的 MAC 地址不会改变 PCI 设备本身；热插拔基本不可行。请详细查看 `readlink /sys/class/net/eth0/device`：

```
(vadbvaz) cc02v014382:/etc/udev/rules.d #
# readlink /sys/class/net/eth0/device
../../../0000:13:00.0
```
